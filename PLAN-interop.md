# Klar — Kira Interop Plan

## Overview

Enable Klar programs to import and call Kira libraries as easily as calling any other Klar module. Today, Kira can generate C headers and Klar extern blocks, but Klar has no awareness of Kira — the developer must manually paste extern blocks and manage linking. This plan adds Klar-side support so the interop is automatic and AI-agent-friendly.

Companion plan: `~/Fun/Kira/PLAN-interop.md` (Kira-side work).

Reference: [DESIGN.md](DESIGN.md) section "C Interoperability", [docs/advanced/ffi.md](docs/advanced/ffi.md).

Current status: Phase 0 not started.

## Phase 0: Consume Kira Type Manifests

**Goal:** Klar can read a Kira type manifest (JSON) and auto-generate the extern block, removing the manual copy-paste step.
**Estimated Effort:** 2 days

### Deliverables
- `klar import-kira <manifest.json>` CLI command
- Auto-generated `.kl` file with extern declarations
- Manifest parser in the Klar toolchain

### Tasks
- [ ] Define the Kira type manifest JSON schema from Klar's perspective — document what Klar needs: function name, parameter names and types, return type, ADT definitions. Coordinate with Kira's `PLAN-interop.md` Phase 4 to ensure the format is agreed. Tests should cover: schema handles primitives, strings, ADTs, void returns, and functions with zero parameters.
- [ ] Implement manifest parser in `src/main.zig` or a new `src/interop/` module — read the JSON file, parse function signatures, and map Kira types to Klar FFI types (i32→i32, f64→f64, string→CStr, bool→Bool, void→void, ADTs→extern struct). Tests should cover: parse a manifest with 5 functions of varying signatures, produce correct internal representation.
- [ ] Implement `klar import-kira <manifest.json> -o <output.kl>` command — read the manifest, generate a `.kl` file containing the extern block and any required extern struct/enum definitions. Tests should cover: generated file is valid Klar syntax, compiles without errors when included in a project.
- [ ] Add the generated file to the module resolution path — when a Klar source file does `import kira_mylib`, the compiler finds the generated `.kl` file. Tests should cover: import resolves, functions from the extern block are callable.

### Testing Strategy
Integration test: take a Kira manifest (hand-written or generated by Kira), run `klar import-kira`, compile the output, verify it type-checks. Compare generated extern block against the expected output for each type mapping.

## Phase 1: ADT Consumption

**Goal:** Klar can work with Kira algebraic data types through extern structs and pattern matching on tags.
**Estimated Effort:** 3 days

### Deliverables
- `extern struct` and `extern enum` generation for Kira ADTs
- Helper functions for tag checking and data access
- Documentation in `docs/advanced/kira-interop.md`

### Tasks
- [ ] Generate `extern enum` for ADT tags from the manifest — each Kira sum type produces an enum with variant names as values. Follow Klar's existing `extern enum` convention (explicit integer values). Tests should cover: `type Shape = Circle(f64) | Rectangle(f64, f64)` produces `extern enum ShapeTag: i32 { Circle = 0, Rectangle = 1 }`.
- [ ] Generate `extern struct` for ADT data layout — match the C struct layout defined in Kira's `docs/design/adt-interop.md`. Include the tag field and data union. Tests should cover: struct field offsets match what Kira's C backend produces, verified by compiling both sides and comparing `sizeof`.
- [ ] Generate safe wrapper functions for ADT access — for each variant, emit a Klar function that checks the tag and returns an optional of the payload. Example: `fn as_circle(shape: ref Shape) -> ?f64`. Tests should cover: `as_circle` returns `Some(radius)` for Circle, `None` for Rectangle.
- [ ] Generate match-friendly helpers — emit a function that returns the tag enum value, enabling Klar `match` on the tag. Tests should cover: `match shape_tag(s) { ShapeTag.Circle => ..., ShapeTag.Rectangle => ... }` compiles and dispatches correctly.
- [ ] Write `docs/advanced/kira-interop.md` — document how to use Kira types from Klar, with examples of calling functions and matching on ADTs.

### Testing Strategy
End-to-end test: Kira library exports a function returning `Shape`, Klar program calls it, matches on the tag, extracts the payload, prints it. Verify correct output for each variant.

### Phase 1 Readiness Gate
Before Phase 2, these must be true:
- [ ] Klar can consume all Kira primitive types and ADTs
- [ ] Tag-based matching works for sum types
- [ ] Record types are accessible as extern structs
- [ ] Documentation covers the full workflow

## Phase 2: String Interop

**Goal:** Strings cross the Kira-Klar boundary safely with clear ownership.
**Estimated Effort:** 2 days

### Deliverables
- String conversion helpers in generated interop code
- Ownership documentation matching Kira's `docs/design/interop-memory.md`
- No memory leaks in string-heavy interop

### Tasks
- [ ] Generate CStr wrappers for Kira string parameters — when calling a Kira function that takes `string`, the generated Klar wrapper calls `.as_cstr()` on the Klar string. Tests should cover: passing a Klar string to a Kira function works, Kira receives correct content.
- [ ] Generate string conversion for Kira string returns — when a Kira function returns `string`, the generated Klar wrapper converts the `CStr` result to a Klar `string` via `.to_string()`. Tests should cover: Kira returns a string, Klar receives it as a proper owned string, can concatenate and print it.
- [ ] Generate `kira_free` extern declaration — include in the generated extern block so Klar can free Kira-allocated memory when needed (for pointer-based returns). Tests should cover: extern declaration compiles, calling it on a Kira-allocated pointer doesn't crash.
- [ ] Document string ownership rules in `docs/advanced/kira-interop.md` — add a section: strings passed to Kira are borrowed for the call duration, strings returned from Kira are owned by Klar after conversion. Tests should cover: documentation is accurate and matches runtime behavior.

### Testing Strategy
Stress test: Klar program calls a Kira string function 10,000 times in a loop. Verify no memory growth (RSS stays stable). Round-trip test: Klar sends "hello", Kira returns "HELLO", Klar verifies equality.

## Phase 3: Build Integration

**Goal:** Klar's build process can automatically build Kira dependencies and link them.
**Estimated Effort:** 3 days

### Deliverables
- `kira` dependency type in Klar project configuration
- Automatic invocation of `kira build --lib` during Klar build
- Automatic import of generated extern blocks

### Tasks
- [ ] Define Kira dependency format in Klar's build configuration — a project can declare a Kira dependency with a path to the Kira project root. Format: `[[deps.kira]]` section with `name` and `path` fields (or `git` for remote). Tests should cover: config parser accepts the new section, rejects malformed entries.
- [ ] Implement build orchestration — during `klar build`, detect Kira dependencies, invoke `kira build --lib` for each, copy the `.a` and `.kl` files to the Klar build directory, add `-l` flags to the link step. Tests should cover: changing a Kira source file triggers rebuild, unchanged sources use cached `.a`.
- [ ] Auto-import generated extern blocks — the generated `.kl` file is added to the compilation unit automatically, so the user just writes `import kira_mylib` without managing files. Tests should cover: import works without manual file placement, functions are callable.
- [ ] Add `klar clean` support for interop artifacts — remove generated `.kl`, `.h`, and `.a` files on clean. Tests should cover: clean removes interop artifacts, rebuild regenerates them.
- [ ] Document the build integration in `docs/advanced/kira-interop.md` — add a section showing the project layout, config format, and build workflow. Include AI agent instructions.

### Testing Strategy
Full integration test project: directory with `klar-project/` and `kira-lib/` subdirectories. `klar build` in the Klar project compiles both, links them, produces a working binary. Verify: initial build works, incremental rebuild after Kira change works, clean + rebuild works.

### Phase 3 Readiness Gate
Before declaring interop complete, these must be true:
- [ ] A Klar project can declare a Kira dependency and build with zero manual steps
- [ ] Type mapping covers all primitive types, strings, and ADTs
- [ ] Memory safety is maintained across the boundary (no leaks, no use-after-free)
- [ ] An AI agent can follow the documentation to set up a cross-language project

## Phase 4: Effect Awareness (Stretch)

**Goal:** Klar's tooling surfaces Kira's purity information so developers and AI agents know which imported functions are side-effect-free.
**Estimated Effort:** 1 day

### Deliverables
- Effect annotations in the type manifest
- Comments or meta annotations in generated Klar code indicating purity

### Tasks
- [ ] Consume effect information from the Kira manifest — the manifest includes an `effect` boolean per function. Tests should cover: manifest with mixed pure/effect functions is parsed correctly.
- [ ] Emit purity comments in generated Klar extern blocks — add `// pure` or `// effect` comments above each function declaration. Tests should cover: comments appear on correct functions.
- [ ] (Optional) Emit Klar `meta` annotations if Klar's meta system supports custom annotations — `@meta(pure)` on pure functions. Tests should cover: meta annotations parse, are queryable via Klar's meta reflection.

### Testing Strategy
Verify generated comments/annotations match the Kira source declarations. No runtime behavior change — this is documentation-only.

## Risk Register

| Risk | Impact | Likelihood | Mitigation |
|------|--------|------------|------------|
| Kira manifest format changes during development | Medium — regeneration needed | Medium | Version the manifest schema; Klar parser checks version field |
| ADT struct layout disagreement between Kira and Klar | High — runtime crashes, data corruption | Medium | Shared test suite that compiles both sides and compares `sizeof` and field offsets |
| `kira` CLI not available on build machine | High — blocks build | Low | Fail with clear error message; document `kira` as a build dependency |
| Incremental build cache invalidation misses | Low — stale artifacts | Medium | Conservative default: rebuild on any Kira source change; optimize later with content hashing |
| Klar's module system doesn't support generated files cleanly | Medium — awkward workarounds | Medium | Prototype early in Phase 0; if blocked, use explicit `import "path"` syntax instead of module names |
